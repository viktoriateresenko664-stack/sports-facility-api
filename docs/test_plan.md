# Test Plan — Sports Facilities Monitoring API

## 1. Цель тестирования
Подтвердить, что backend:
- корректно реализует ролевую модель и ownership;
- стабильно обслуживает web/mobile/desktop сценарии;
- безопасно обрабатывает входные данные и доступ;
- сохраняет backward compatibility API-контрактов.

## 2. Область тестирования
### Входит в область
- Auth и JWT жизненный цикл;
- RBAC и ownership;
- BFF endpoint-ы для трёх клиентов;
- инженерные задачи и статусы;
- отчёты (upload/generate/download/preview/jobs);
- realtime-каналы (WebSocket/SSE);
- CORS/security ограничения;
- обработка ошибок и коды ответа.

### Не входит в область
- внешний UI/верстка клиентов;
- нагрузочное тестирование уровня production-scale;
- хаос-тесты инфраструктуры cloud-провайдера.

## 3. Типы тестов
- smoke;
- functional;
- integration API;
- security checks (включая Burp Suite сценарии);
- regression.

## 4. Тестовое окружение
- backend FastAPI;
- PostgreSQL (тестовая база, не production);
- Redis/Celery (для фоновых задач);
- отдельные тестовые JWT для ролей.

Обязательное правило: не использовать production-данные для автотестов.

## 5. Входные критерии
- миграции применены;
- сервис поднимается без ошибок;
- тестовые роли доступны;
- dev/prod env флаги установлены согласно сценарию.

## 6. Выходные критерии
- критические и высокие дефекты отсутствуют;
- все обязательные тест-кейсы пройдены;
- регрессии ключевых сценариев нет.

## 7. Набор обязательных проверок
## 7.1 Auth
1. `POST /auth/register` — успешная регистрация.
2. `POST /auth/login` — вход пользователя.
3. `POST /auth/employee-login` — вход сотрудника.
4. `POST /auth/refresh` — обновление пары токенов.
5. `POST /auth/logout` — инвалидация refresh.
6. `GET /auth/me` — получение профиля.
7. `POST /auth/me` — обновление профиля.
8. Негатив: невалидный токен -> `401`.

## 7.2 RBAC
1. USER не может открыть `/bff/desktop/*` -> `403`.
2. ENGINEER не может открыть desktop BFF -> `403`.
3. ENGINEER не может открыть `/ws/tasks` и `/ws/sensors` -> отказ.
4. OPERATOR может открыть desktop BFF.
5. CHIEF_ENGINEER может открыть desktop BFF.
6. ENGINEER может открыть `/bff/mobile/tasks`.
7. USER не может открыть `/bff/mobile/tasks` -> `403`.

## 7.3 Ownership
1. USER видит только свои `user_requests`.
2. ENGINEER видит только свои `engineer_tasks`.
3. ENGINEER не может загрузить отчёт по чужой задаче.
4. `GET /reports/jobs/{job_id}` не раскрывает чужой job.

## 7.4 Reports
1. `POST /reports/upload` принимает ключ файла `report_file`.
2. Неверный ключ файла возвращает `422`.
3. `POST /reports/generate-delayed` возвращает `job_id`.
4. `GET /reports/jobs/{job_id}` отдаёт статус в доменной шкале.
5. `GET /reports/{report_id}/download` работает для `uploaded_file`.
6. `GET /reports/{report_id}/download` работает для `generated_text`.
7. `GET /reports/{report_id}/preview` работает для поддерживаемых типов.

## 7.5 Security
1. Без токена -> `401`.
2. Неправильная роль -> `403`.
3. Несуществующий ресурс -> `404`.
4. Бизнес-конфликт -> `409`.
5. Невалидное тело -> `422`.
6. SQL injection payload не должен ломать query.
7. XSS payload не должен возвращаться опасным HTML.

## 8. Матрица критичных бизнес-сценариев
1. USER создаёт заявку -> OPERATOR назначает инженера -> ENGINEER выполняет задачу -> отчёт появляется в desktop.
2. CHIEF_ENGINEER создаёт задачу из мониторинга датчика -> ENGINEER выполняет -> отчёт доступен для desktop.
3. Realtime: изменение статуса задачи отражается в desktop/mobile без ручного polling.

## 9. Нефункциональные проверки
- корректность CORS в production-конфигурации;
- стабильность фоновых задач (job status flow);
- логирование действий без утечки секретов;
- корректная деградация при недоступности внешних зависимостей.

## 10. Проверка безопасности через Burp Suite (минимальный чек-лист)
1. Попытка IDOR по `task_id`, `report_id`, `job_id`.
2. Подмена ролей/subject в payload.
3. Массовое присваивание запрещённых полей.
4. Проверка rate limit/bruteforce на login endpoint-ах.
5. Попытка загрузки опасного файла в отчёт.
6. Проверка refresh token reuse.

## 11. Артефакты тестирования
- отчёт автотестов `pytest`;
- список пройденных smoke-сценариев;
- перечень найденных дефектов и статус исправления;
- журнал security-check результатов.

## 12. Критерии готовности к защите
- все обязательные сценарии разделов 7 и 8 воспроизводимы;
- нет несоответствий между документацией и фактическим поведением API;
- можно объяснить любой endpoint по цепочке: роль -> ownership -> данные -> результат.
