# Sports Facilities Monitoring API — Полная инструкция по работе приложения

## 1. Назначение системы
Система предназначена для мониторинга спортивных объектов, управления заявками и работой инженеров, а также подготовки и хранения технических отчётов.

Единый backend API обслуживает три клиента:
- web-клиент для конечного пользователя;
- mobile-клиент для инженера;
- desktop-клиент для оператора и инженера по эксплуатации.

## 2. Роли и зоны ответственности
### USER (сайт)
- регистрируется и входит в систему;
- создаёт заявки по объектам;
- видит только свои заявки и свои данные.

### ENGINEER (мобильный клиент)
- видит только назначенные на него задачи;
- переводит задачу по жизненному циклу (start/finish);
- загружает отчёты только по своим задачам;
- скачивает шаблон отчёта.

### OPERATOR (desktop)
- обрабатывает пользовательские заявки;
- назначает инженера на заявку;
- контролирует списки заявок, задач, отчётов, сотрудников, логи.

### CHIEF_ENGINEER (desktop, инженер по эксплуатации)
- имеет расширенные права desktop;
- может работать с мониторингом;
- может создавать инженерные задачи напрямую из контекста датчика/оборудования;
- контролирует отчётность и состояние инфраструктуры.

## 3. Основные сущности
- `sports_facilities` — спортивные объекты;
- `equipment` — оборудование объекта;
- `sensors` и `sensor_data` — датчики и их показания;
- `user_requests` — заявки пользователей;
- `engineer_tasks` — задачи инженерам;
- `engineer_reports` — отчёты инженеров;
- `background_jobs` — фоновые задания;
- `domain_events` — доменные события;
- `system_action_log` — журнал действий.

## 4. Жизненный цикл бизнес-процесса
### 4.1 Заявка пользователя
1. Пользователь создаёт заявку.
2. Заявка получает статус `CREATED`.
3. Оператор/главный инженер назначает инженера.
4. При назначении создаётся `engineer_task`, а заявка становится `ASSIGNED`.
5. После старта задачи заявка переходит в `IN_WORK`.
6. После завершения задачи заявка переходит в `COMPLETED`.

### 4.2 Статусы заявок
- `CREATED`
- `ACTIVE`
- `ASSIGNED`
- `IN_WORK`
- `COMPLETED`
- `CANCELLED`

### 4.3 Статусы задач инженера
- `CREATED`
- `ACTIVE`
- `COMPLETED`
- `CANCELLED`

Разрешённые переходы:
- `CREATED -> ACTIVE`
- `ACTIVE -> COMPLETED`
- `ACTIVE -> CANCELLED`

Некорректный переход возвращает `409 Conflict`.

## 5. Клиентские сценарии
### 5.1 WEB (USER)
- Вход/профиль.
- Получение объектов для выбора.
- Создание заявки.
- Просмотр только своих заявок.
- Просмотр карты объектов через BFF.

### 5.2 MOBILE (ENGINEER)
- Вход инженера по ключу и паролю.
- Получение списка своих задач.
- Начало/завершение задачи.
- Скачивание шаблона отчёта.
- Загрузка отчёта по задаче.
- Получение realtime-событий по задачам (SSE).

### 5.3 DESKTOP (OPERATOR/CHIEF_ENGINEER)
- Мониторинг панели.
- Работа с заявками и назначениями.
- Просмотр отчётов и скачивание/предпросмотр.
- Просмотр логов и сотрудников.
- Realtime через WebSocket.

Дополнительно для CHIEF_ENGINEER:
- создание задачи напрямую по датчику/оборудованию (без пользовательской заявки).

## 6. Отчёты: как работает сейчас
Поддерживаются два типа отчёта:
- `uploaded_file` — отчёт загружен файлом;
- `generated_text` — отчёт сохранён как текст.

Скачивание и предпросмотр:
- для `uploaded_file` используется оригинальный файл;
- для `generated_text` backend формирует текстовую выдачу (`.txt`) для download/preview.

Это важно для desktop: один и тот же реестр отчётов корректно работает с обоими типами источников.

## 7. BFF слой (адаптация под UI)
Система использует BFF-разделение:
- `/bff/web/*` — компактные DTO под web;
- `/bff/mobile/*` — DTO под mobile-задачи;
- `/bff/desktop/*` — агрегированные данные для desktop.

Это позволяет:
- не переносить бизнес-логику в клиенты;
- формировать разные ответы под разные UX-сценарии;
- централизованно контролировать права доступа.

## 8. Realtime и синхронизация данных
### Desktop
- `ws/tasks` — события по задачам (`TASK_CREATED`, `TASK_ASSIGNED`, `TASK_STARTED`, `TASK_COMPLETED`, `REPORT_READY`, `TASK_UPDATED`);
- `ws/sensors` — поток обновления датчиков.

Доступ: только `OPERATOR`, `CHIEF_ENGINEER`.

### Mobile
- SSE поток: `/bff/mobile/events/stream`.

Доступ: только `ENGINEER`, с фильтрацией по ownership.

## 9. Безопасность: как реализована по факту
### 9.1 Авторизация и роли
- JWT access/refresh;
- role-based доступ на каждом защищённом endpoint;
- ownership-проверки на уровне backend.

### 9.2 Ownership
- `user_id`, `employee_id`, `engineer_id` берутся из JWT/серверной модели;
- клиент не может назначить владельца вручную;
- попытка доступа к чужому ресурсу блокируется.

### 9.3 Валидация и защита от mass assignment
- DTO с `extra=forbid`;
- явный маппинг полей на сервере.

### 9.4 SQL Injection
- ORM для команд;
- raw SQL только с bind-параметрами.

### 9.5 XSS
- серверная санитаризация пользовательского текста;
- безопасный возврат данных для frontend-рендера.

### 9.6 CORS
- в production разрешены только whitelisted origins;
- wildcard `*` запрещён в production;
- regex для origin в production отключается.

### 9.7 Брутфорс и rate limit
- ограничение попыток логина;
- блокировка после серии неуспешных попыток;
- общее ограничение запросов.

## 10. Асинхронность, очередь, eventual consistency
Тяжёлые операции (например, генерация отчёта) выполняются в фоне:
- API сразу возвращает `job_id` и текущий статус;
- результат доступен через статус endpoint;
- read-модель и realtime могут обновляться с небольшой задержкой.

Это нормальная eventual consistency модель.

## 11. Политика методов API
Текущая реализация ориентирована на:
- `GET` — чтение;
- `POST` — создание/действия/обновления.

Новые контракты добавляются в этой же политике без ломки текущих клиентов.

## 12. Ошибки и их смысл для клиентов
- `401` — токен отсутствует/невалиден;
- `403` — роль не подходит или действие запрещено;
- `404` — ресурс не существует или скрыт ownership-проверкой;
- `409` — бизнес-конфликт (например, недопустимый переход статуса);
- `422` — ошибка валидации входных данных.

## 13. Что обязательно проверить перед демонстрацией
1. Вход по ролям USER / ENGINEER / OPERATOR / CHIEF_ENGINEER.
2. Создание заявки пользователем и назначение инженера на desktop.
3. Начало/завершение задачи инженером.
4. Появление отчёта в desktop реестре.
5. Работу download/preview отчётов.
6. Карта объектов с координатами.
7. Realtime обновления задач.
8. Ограничения доступа (чужие данные недоступны).

## 14. Короткий ответ на типовой вопрос преподавателя
“Как связаны BFF, ownership и CORS в одной функции?”

Пример на endpoint списка задач mobile:
1. Клиент mobile вызывает BFF endpoint.
2. CORS пропускает только разрешённый origin.
3. JWT проверяется, роль должна быть `ENGINEER`.
4. Backend извлекает `employee_id` из токена.
5. Запрос в БД фильтруется по `assigned_engineer_id = employee_id`.
6. Возвращается DTO mobile-формата (без лишних полей desktop/web).

Итог: один вызов проходит сразу три уровня контроля — сеть (CORS), роль (RBAC), владелец данных (ownership).
